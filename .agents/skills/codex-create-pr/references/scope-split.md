# スコープと分割（要約）

## 目的

- 1つのPRは1つの主目的に絞り、レビューしやすさを最優先する。

## 分割すべき強いシグナル

- 目的が2つ以上ある（機能追加とバグ修正が混在など）。
- リファクタリングや整形と挙動変更が混在している。
- 互いに独立な変更が同じPRに入っている。

## サイズ・レビュー負荷の目安（ソフト）

- 差分が400行超、または10ファイル超、または4領域以上にまたがる場合は分割検討。
- ここでの「領域」はトップレベルフォルダやサブシステム単位を指す。
- 一回のレビューで60分以上かかりそうなら分割検討。
- 2つ以上該当する場合は分割提案を強める。

## 分割の進め方

1. 前提となる下地変更を切り出す。
2. 依存順を決めて小さいPRから出す。
3. それぞれのPRに目的と検証範囲を明記する。

## 分割できない場合

- 1つにまとめる理由と影響範囲をPR説明文に明記する。
