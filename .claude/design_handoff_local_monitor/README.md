# Handoff: Local Monitor 再設計（Copilot Agent Observability — Local Ingestion Monitor UI）

最終更新: 2026-07-03 / デザイン・IA 確定版

---

## 1. Overview

GitHub Copilot Chat / CLI / Codex App の OpenTelemetry データをローカル収集する
**Local Ingestion Monitor**（`http://127.0.0.1:4320`）の UI 全面再設計。

再設計の狙い:

- 最優先ユーザー = **開発者**、最重要シナリオ = **token コストの把握・削減**
- 俯瞰（概要）→ 一覧（トレース）→ 詳細（フロー）→ 生データ（raw）の自然なドリルダウン
- キャッシュ効率を第一級の観点として常設
- エラー trace の調査を「一覧 → 詳細 → 原因の手がかり」まで一画面で完結
- GitHub Copilot（ローカル SDK 経由）による trace 解析チャットを組み込み

## 2. About the Design Files

このバンドル内のファイルは **HTML で作られたデザインリファレンス**（プロトタイプ）であり、
そのまま本番に流用するコードではない。タスクは、これらのデザインを
**対象コードベース（.NET LocalMonitor の静的 HTML/JS UI、`wwwroot/`）の既存環境・既存パターンで再現する**こと。
フレームワークが未確定の部分は、既存の LocalMonitor 実装（vanilla JS + sanitized DOM 生成）に合わせるのが第一候補。

- `ローカルモニター再設計.dc.html` — 全ターンのデザインカンバス。**TURN 4（最上部）+ TURN 3 + TURN 2 の 2a-3 + TURN 1 の A1/A2 が確定仕様**。TURN 1 の「1b 台帳型」案は不採用（参考のみ）
- `support.js` — プロトタイプ表示用ランタイム（実装対象ではない）

## 3. Fidelity

**High-fidelity**。色・タイポグラフィ・余白・角丸・状態表現は最終値として扱い、ピクセル単位で再現する。
ただし**表示されているデータはすべて合成ダミー**（trace 名、トークン数、時刻等）。実データバインドは §9 参照。

## 4. 確定した IA（情報アーキテクチャ）

- 採用方向: **Console 型** — 左サイドバー常駐 + master-detail（TURN 1 案 1a。1b 台帳型は不採用）
- ナビゲーションは **2 項目のみ**: 「概要」「トレース」（トレースには件数バッジ）
- **「診断」はナビから削除**。サイドバー最下部の受信ステータスバッジ → クリックでポップオーバー → 「詳細診断を開く」で診断画面へ、の段階的動線に格納
- 画面一覧（実装対象）:
  1. **概要**（ダッシュボード）
  2. **トレース一覧**（master-detail、右にプレビュー）
  3. **トレース詳細**（フロー / waterfall 切替 + キャッシュ列）
  4. **スパンインスペクタ**（詳細画面内の右パネル、クリックで開く）
  5. **エラー解析モード**（エラー trace の詳細画面バリアント）
  6. **Copilot 解析ドロワー**（詳細画面上の右ドロワー）
  7. **診断**（ポップオーバー経由でのみ到達）

ドリルダウン: 概要のあらゆる数値・行 → トレース一覧（フィルタ適用済み）→ 行選択でプレビュー → 「詳細を開く」→ トレース詳細 → スパンクリックでインスペクタ / raw。

### 6.5 エラー trace の解析モード

参照: canvas「4a-2 エラー解析」。エラーを含む trace の詳細画面バリアント。

- status ピルが「● エラー · 異常終了」（赤）
- タイトル下に**エラー要約ストリップ**（背景 `rgba(224,94,80,0.07)`、境界 `#4a2a2a`）: 「!」丸 + 「**エラー 2件** — 1件は回復済み(ターン4 str_replace)、1件が原因でトレースが異常終了(ターン7 llm 呼出)」+ 右端「最初のエラーへ」「エラーのみ表示中」ボタン
- フローは**「エラーのみ(前後の文脈つき)」フィルタ ON** がデフォルト:
  - 正常区間は破線カードに折りたたみ「ターン1〜3 · 正常 — 6 スパンを折りたたみ 展開」
  - **回復済みエラー**: ツールカードが琥珀（背景 `rgba(224,170,78,0.06)`、境界 `#5a4a22`）+ 「✕ 失敗 · match not found」+ 横に「回復済み → ターン5 で再試行」ピル。続けて再試行スパン（「再試行 · 成功」緑）
  - **未回復エラー**: 赤カード（境界 `#e05e50` + glow）「ターン7 · LLM 呼出 失敗」+ 「✕ context_length_exceeded — retry 2回とも失敗」+ 「解析パネルに表示中 →」。マーカーも赤大
  - 終端: ✕ 赤円「異常終了 — トレースは未完了のまま終了」+ 時刻
- 右列はキャッシュ列の代わりに**エラー解析パネル 3 カード**:
  1. **エラー一覧**（件数付き）: 行カード = ●色 + 「str_replace · match not found」+ サブ「ターン4 · 3s」+ 右端に「回復済み」（琥珀）/「未回復」（赤）ピル。クリックで選択（選択行は赤ハイライト）→ 詳細カードとフローの該当スパンが連動
  2. **エラー詳細**（選択中エラー）: span id / 例外 type + message の Mono 赤ブロック（例 `prompt is too long: 131,449 tokens > 128,000 maximum`）/ kv 3 行（モデル・発生・リトライ「+21s 失敗 · +27s 失敗」）/ アクション「raw を開く」「Copilot で解析」
  3. **原因の手がかり — 入力トークンの推移**: ターン別棒グラフ + **上限 128K の赤破線**、超過ターンの棒は赤。解説文 + 「該当スパンを見る」リンク

### 6.6 Copilot 解析ドロワー

参照: canvas「3a-2 Copilot ドロワー」（形態）+「2a-2 Copilot 解析チャット」（連動仕様）

- 起動: ヘッダー常設「Copilot で解析」/ スパンホバーの「このスパンを Copilot に聞く」/ エラー詳細の「Copilot で解析」
- **別画面ではなく右ドロワー**（幅 472px、絶対配置でキャッシュ列の上にスライドイン。背面のフローは opacity 0.55 で見えたまま。背景 `#12151c`、左境界 `#4a4080`、影 `-24px 0 64px rgba(0,0,0,0.55)`）
- ヘッダー: Copilot アイコン丸 + 「GitHub Copilot で解析」+ **「ローカル SDK 経由 · raw はローカルから出ません」**（データ境界の明示、必須コピー）+ trace id チップ + ✕
- フロー: 観点（例「トークン」）を選んで解析実行 → 実行チップ「観点「トークン」で解析を実行 — 3.2s」→ **所見カード**(番号付き所見 + 「該当スパンを表示」リンク + 「推定削減余地 約 18K (−14%)」琥珀ボックス) → 以降**チャットで追い質問**できる
- 所見・回答内の「該当スパンを表示」で**背面フローの該当スパンを強調表示**（紫系ハイライト）し、クリックでその位置へスクロール
- サジェスト質問チップ（紫アウトラインピル）: 「このトレースの削減余地は?」「キャッシュ作成を増やすべき?」「似た傾向の trace を探す」
- 下部: 入力欄「さらに質問…」+ ↑ 送信ボタン（青）
- バックエンド: raw OTLP ペイロードを .NET Copilot SDK でローカル解析。raw は外部送信しない

### 6.7 診断（詳細画面）

参照: canvas「A4 · 診断」。ポップオーバーの「詳細診断を開く」からのみ到達。

- 見出し「診断」+ 「受信できます — ready」緑ピル + 右端 `GET /health/ready → 200`
- **パイプライン図**: 4 カード（① 受信 OTLP/HTTP → ② 書き込み キュー→raw store → ③ Projection raw→表示用データ → ④ 表示 UI/API 提供）を「→」で接続。各カードに OK/NG + Mono の実測値 2 行
- **コンポーネント確認テーブル**: Loopback bind / DB / Migration / Writer / Projection worker / Ingestion queue — 各行に役割説明文 + 実測値 + OK バッジ。ヘッダー注記「問題があった時だけ読む場所 — 正常時は全行 OK」
- しきい値: projection 遅延 < 30s / projection 残 < 500 / span projection 残 < 2000 / 受信 stall < 60s

## 7. Interactions & Behavior まとめ

- 遷移: 概要 KPI・行 → フィルタ済み一覧 / 一覧行クリック → プレビュー更新（遷移なし）/ 「詳細を開く」→ 詳細 / 詳細の前・次で隣接 trace へ
- フロー⇄waterfall 切替はセグメントコントロール。**選択スパン・エラーのみフィルタ・インスペクタ状態は切替をまたいで保持**
- スパンクリック → インスペクタ（§6.4 のルール）。エラー一覧クリック → エラー詳細 + フロー該当スパン連動
- ホバー: テーブル行 `#212736` / フローカード境界色変化 / スパンに Copilot ピル
- トランジション: 150ms ease-out（基本）、ドロワー/ポップオーバーはスライド+フェード 200ms。`prefers-reduced-motion: reduce` で 0ms
- キーボード: Esc（インスペクタ / ドロワー / ポップオーバーを閉じる）、テーブルソート・フィルタは Tab/Enter 操作可能
- ローディング/空状態: データは即時表示（演出なし）。受信ゼロ時はステータスバッジが案内役

## 8. State Management

- グローバル: 受信ヘルス（正常/警告/異常 + パイプライン 4 段の状態）、trace 件数
- 概要: 期間（今日/7日/30日）
- 一覧: 検索文字列、モデル/状態/期間フィルタ、ソートキー（既定: トークン降順）、選択 trace、ページング位置
- 詳細: 現在 trace id、ビュー（flow | waterfall、既定 flow）、エラーのみフィルタ（エラー trace では既定 ON）、折りたたみ状態、選択スパン id（= インスペクタ開閉）、インスペクタタブ（整形 | raw）
- ドロワー: 開閉、観点、解析結果、チャット履歴（trace 単位）、ハイライト中スパン
- URL に trace id / ビュー / 選択スパンを反映させ、リロード・共有に耐えること（推奨）

## 9. データ要件（表示に必要な導出値）

- trace 単位: プロンプト冒頭、モデル、クライアント、開始時刻、所要、ターン数、ツール呼出数、status（正常 / エラー(回復済みのみ) / 異常終了）、トークン内訳（cache read / cache create / uncached input / output）、**実効入力換算 = cache_read×0.1 + uncached_input**
- ターン(llm span)単位: 意図ラベル（プロンプト/応答から生成 or span 名）、tok 合計、cache%、所要、メッセージロール別サイズ
- tool span 単位: ツール名+主要引数の 1 行表記、所要、結果 status、結果サイズのトークン換算、親ターン、**並行グループ判定（開始時刻が重なる同一親の tool span 群）**
- エラー: exception.type / message、リトライ回数と結果、**回復済み判定（同種操作の後続成功）**
- 集計: 日/期間別トークン、モデル別内訳、キャッシュ効率、高コスト TOP5、時間帯分布
