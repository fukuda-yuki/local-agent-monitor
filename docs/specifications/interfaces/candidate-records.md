# Candidate Record Interfaces

Candidate records are deterministic outputs used to explain possible workflow failures and propose next actions.
They are recommendation data only; they do not apply file changes, create commits, push branches, or prove improvement effects.

## Diagnosis Candidate

Produced by:

```text
config-cli generate-diagnosis-candidates <measurements.csv|measurements.json> [--raw <raw-store.db|raw-otlp.json>] [--include-sensitive-content] [--sensitive-output-dir <dir>] [--csv <output.csv>] [--json <output.json>]
```

CSV column order and JSON property names:

```text
diagnosis_candidate_id,trace_id,source_record_ref,rule_id,failure_category_id,anti_pattern_id,severity,recommended_improvement_target,evidence_summary,evidence_ref,content_included,sensitive_bundle_path,confidence,required_human_checks,candidate_status
```

| Field | Semantics |
| --- | --- |
| `diagnosis_candidate_id` | Stable candidate id generated by the CLI. |
| `trace_id` | Source trace reference when available. |
| `source_record_ref` | Reference to the source measurement or raw evidence record. |
| `rule_id` | Deterministic rule that emitted the candidate. |
| `failure_category_id` | Product-level failure category. |
| `anti_pattern_id` | Optional anti-pattern classification. |
| `severity` | Candidate severity such as `low`, `medium`, or `high`. |
| `recommended_improvement_target` | Area that should be inspected or improved. |
| `evidence_summary` | Sanitized evidence summary. |
| `evidence_ref` | Reference to evidence, not the evidence body itself. |
| `content_included` | `true` only when sensitive evidence extraction was explicitly enabled. |
| `sensitive_bundle_path` | Local-only path to an opt-in sensitive bundle. Do not commit or publish. |
| `confidence` | Rule confidence label. |
| `required_human_checks` | Human review checks needed before acting on the candidate. |
| `candidate_status` | Candidate lifecycle status. |

## Improvement Candidate

Produced by:

```text
config-cli generate-improvement-candidates <diagnosis-candidates.csv|diagnosis-candidates.json> [--csv <output.csv>] [--json <output.json>]
```

CSV column order and JSON property names:

```text
improvement_candidate_id,source_diagnosis_candidate_id,trace_id,failure_category_id,anti_pattern_id,severity,improvement_target,proposal_title,proposal_summary,proposed_change_kind,evidence_ref,sensitive_bundle_path,candidate_status
```

| Field | Semantics |
| --- | --- |
| `improvement_candidate_id` | Stable improvement candidate id generated by the CLI. |
| `source_diagnosis_candidate_id` | Diagnosis candidate that led to this proposal. |
| `trace_id` | Source trace reference when available. |
| `failure_category_id` | Failure category inherited from diagnosis. |
| `anti_pattern_id` | Optional anti-pattern inherited from diagnosis. |
| `severity` | Severity inherited or normalized for prioritization. |
| `improvement_target` | Target area for the proposed improvement. |
| `proposal_title` | Short proposal title. |
| `proposal_summary` | Sanitized proposal explanation. |
| `proposed_change_kind` | Type of change being suggested. |
| `evidence_ref` | Sanitized evidence reference. |
| `sensitive_bundle_path` | Local-only path to an opt-in sensitive bundle. Do not commit or publish. |
| `candidate_status` | Candidate lifecycle status. |

## Auto-decision Record

Produced by:

```text
config-cli generate-auto-decisions <improvement-candidates.csv|improvement-candidates.json> [--csv <output.csv>] [--json <output.json>]
```

CSV column order and JSON property names:

```text
auto_decision_id,source_improvement_candidate_id,source_diagnosis_candidate_id,trace_id,decision_status,decision_rule_id,decision_reason,confidence,blocking_risk_checks,sensitive_content_included,sensitive_bundle_path,implementation_target,next_action
```

| Field | Semantics |
| --- | --- |
| `auto_decision_id` | Stable decision record id generated by the CLI. |
| `source_improvement_candidate_id` | Improvement candidate being evaluated. |
| `source_diagnosis_candidate_id` | Original diagnosis candidate. |
| `trace_id` | Source trace reference when available. |
| `decision_status` | Recommendation status such as proceed, hold, or needs review. |
| `decision_rule_id` | Deterministic rule that created the decision. |
| `decision_reason` | Sanitized explanation of the decision. |
| `confidence` | Decision confidence label. |
| `blocking_risk_checks` | Checks that block automatic progression. |
| `sensitive_content_included` | `true` when the source candidate was generated with sensitive content. |
| `sensitive_bundle_path` | Local-only path to an opt-in sensitive bundle. Do not commit or publish. |
| `implementation_target` | Target area if a human later chooses to implement the proposal. |
| `next_action` | Suggested next manual action. |

## Sensitive Bundle Boundary

Candidate records with a populated `sensitive_bundle_path` are local runtime artifacts.
They are not repository-safe, even when the evidence body is stored outside the CSV or JSON.

Repository-safe sharing should use:

- sanitized candidate rows with `sensitive_bundle_path` removed, or
- [Dashboard dataset](dashboard-dataset.md) rows that expose only `sensitive_bundle_present`.
