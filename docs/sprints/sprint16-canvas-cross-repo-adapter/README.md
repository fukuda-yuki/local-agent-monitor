# Sprint16: Canvas Cross-Repo Adapter Packaging

Sprint16 makes the GitHub Copilot app Canvas adapter copyable across
repositories while preserving the Local Monitor boundary established in
Sprint11 and Sprint15.

The extension distribution unit is:

```text
.github/extensions/otel-monitor-canvas/
```

This folder remains the single source of truth. Do not create a second mirror
under another extension directory unless a future spec explicitly requires it.

## Goal

A user can copy the Canvas adapter into either a user-scoped or project-scoped
Copilot extension location, open it against a loopback Local Monitor, and tell
which repository/workspace each trace belongs to without exposing absolute
local paths, raw prompt/response bodies, tool arguments/results, credentials,
tokens, or raw OTLP payloads.

## Controlling guidance

- Current source of truth must be updated before implementation changes that
  alter product behavior, public interfaces, monitor projection schema, or data
  safety policy.
- Canvas action responses remain bounded DTOs.
- The helper page may show sanitized repository/workspace labels and links to
  existing Local Monitor pages, but it must not reimplement the Local Monitor UI.
- The normal raw-default Local Monitor remains allowed. `--sanitized-only` is an
  optional monitor mode, not a Canvas adapter prerequisite.
- The extension must not add new runtime/development dependencies, a
  `package.json`, a lockfile, or `node_modules` in this sprint.
- Live Canvas runtime validation is a final human-gated step. Automated tests
  and contract tests do not replace it.

## Metadata contract

Sprint16 intentionally uses only existing/recommended OTLP resource attributes:

| Local Monitor / Canvas field | Source attribute | Notes |
| --- | --- | --- |
| `repository_name` | `repo.name` | Sanitized display label. |
| `workspace_label` | `workspace.name` | Sanitized display label; not an absolute path. |
| `repo_snapshot` | `repo.snapshot` | Sanitized branch/commit/snapshot label if present. |

Missing metadata displays as `unknown repository` in the Canvas helper page.
Existing projected rows are not automatically backfilled; their new columns
remain null until new telemetry is ingested or the database is regenerated by a
separate explicit operation.

Do not add `repository_full_name`, `workspace_hash`, `git_branch`,
`git_commit_sha`, or `source_kind` in this sprint. There is no selected,
sanitized, existing source for those fields.

## Implementation order

Implement one milestone at a time and commit at each clean boundary:

1. M1 updates source-of-truth docs only.
2. M2 changes Local Monitor projection/API behavior.
3. M3 changes copyable extension packaging assets/docs.
4. M4 changes Canvas helper behavior.
5. M5 records validation, review, live evidence or blocker, and final status.

M2 and M3 touch mostly independent areas, but keep them sequential in this
repository to make review and rollback simpler. M4 depends on both M2's new
metadata fields and M3's copyable-extension framing.

## Acceptance criteria

- `.github/extensions/otel-monitor-canvas/` can be copied as a complete Canvas
  extension folder.
- The folder contains extension-local `canvas.json`, `README.md`, and a
  synthetic `assets/preview.png`.
- Local Monitor trace list, summary highlight trace objects, and trace detail
  surfaces expose only sanitized `repository_name`, `workspace_label`, and
  `repo_snapshot`.
- Canvas helper page shows extension scope, monitor URL/readiness,
  repository/workspace trace labels, a manual repository/workspace filter, and
  `unknown repository` fallback.
- Canvas action responses stay bounded and do not include raw/PII/path/token
  data.
- Loopback-only monitor URL validation, helper-server token protection, and
  existing raw-preview boundaries remain in force.
- User-scope and project-scope live validation is either completed and recorded
  or explicitly blocked with the missing runtime/tool evidence named.
